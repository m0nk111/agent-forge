# Code of Conduct

## Our Pledge

We as contributors and maintainers pledge to making open source participation a harassment-free experience for everyone, regardless of age, body size, disability, ethnicity, gender identity and expression, level of experience, nationality, personal appearance, race, religion, or sexual orientation.

## Our Standards

Examples of behavior that contributes to creating a positive environment include:

- Using welcoming and inclusive language
- Being respectful of differing viewpoints and experiences
- Giving credit where it is due
- Using appropriate language and humor

Examples of unacceptable behavior by participants include:

- The use of sexualized language or imagery
- Telling personal stories without consent
- Repeatedly asking for information or support
- Insulting, demeaning, or derogatory comments
- Publicly shamefully attacking others

## Reporting Behavior

If you experience or witness unacceptable behavior, please notify a project maintainer immediately. Maintainers are listed in the [project README](README.md). Project maintainers will be happy to help address the situation.

## Our Responsibilities

Project maintainers are responsible for clarifying the standards of acceptable behavior and are expected to take appropriate and fair corrective action in response to any unacceptable behavior.

## Enforcement

Any instances of unacceptable behavior may result in a temporary or permanent ban from participating in this project. The decision will be reviewed by the project maintainers.

This Code of Conduct is adapted from the [Contributor Covenant](https://www.contributor-covenant.org/).

# How to Submit Issues for Implementing Changes

When submitting an issue related to implementing changes, please follow these guidelines:

1. **Check for Existing Issues**: Before creating a new issue, search through existing issues to ensure that your problem or question has not already been reported.
2. **Be Specific and Clear**: Provide as much detail as possible about the change you are requesting. This includes the current behavior, expected behavior, and any relevant code snippets.
3. **Use a Descriptive Title**: The title of your issue should clearly describe the change you are requesting.
4. **Provide Context**: Include any relevant context that might help understand the change better, such as the use case or specific requirements.

# How to Submit PRs

When submitting a pull request (PR) to implement changes, please follow these guidelines:

1. **Fork and Clone the Repository**: Fork the repository on GitHub and clone it to your local machine.
2. **Create a New Branch**: Create a new branch from the `main` branch for your changes.
3. **Make Your Changes**: Make the necessary changes in your branch.
4. **Commit Your Changes**: Commit your changes with a descriptive commit message.
5. **Push Your Changes**: Push your changes to your forked repository.
6. **Create a Pull Request**: Go to the original repository on GitHub, select your branch, and create a new pull request. Provide a clear title and description explaining your changes.

    - Run tests: Ensure that all tests pass locally before submitting your PR.
    - Lint your code: Follow the project's linting guidelines to maintain code quality.
    - Update documentation: If your change affects documentation, update it accordingly.

# How to Implement Changes

When implementing changes, please follow these guidelines:

1. **Understand the Requirements**: Ensure you understand the requirements of the change before starting.
2. **Write Clean Code**: Write clean, readable, and maintainable code.
3. **Add Tests**: Add tests for your changes to ensure they work as expected.
4. **Document Your Changes**: Document any changes made in a clear and concise manner.

# How to Submit Issues

When submitting an issue, please follow these guidelines:

1. **Check for Existing Issues**: Before creating a new issue, search through existing issues to ensure that your problem or question has not already been reported.
2. **Be Specific and Clear**: Provide as much detail as possible about the issue you are reporting. This includes any relevant error messages, steps to reproduce the issue, and any other context that might help.
3. **Use a Descriptive Title**: The title of your issue should clearly describe the problem you are encountering.
4. **Provide Context**: Include any relevant context that might help understand the issue better, such as the version of the software you are using or any specific environment in which the issue occurs.

Thank you for contributing to our project!
